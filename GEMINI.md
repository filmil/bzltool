# Gemini Project Instructions

When working on this project, please adhere to the following constraints and
guidelines as defined in the `ai/spec.md` specification:

## Core Technologies
* **Build System:** Use Bazel (`MODULE.bazel`) for all builds and tests.
* **Language:** Use Go.
* **Templating:** Use Go `text/template` for generating configuration files.
* **CLI & TUI:** Use the Cobra library for flag/config management, and
  Bubbletea for TUI elements when used interactively.

## Code Standards & Testing
* **Documentation:** All public APIs must be documented according to Go
  documentation standards (godoc comments).
* **Testing:** Every feature must have at least one unit test confirming its
  functionality. Aim for 100% code path coverage. Every unit test shall include
  multiple test cases that cover regular paths, error paths, and edge cases.
* **Commit Messages:** Use Conventional Commits v1.0.0 for all git commit
  messages (e.g., `feat:`, `fix:`, `docs:`, `test:`).

## Feature Details
* The tool operates on template repositories containing `skills/` and
  `fragments/` directories.
* Fragments are concatenated alphanumerically by path if defined across
  multiple sources.
* `bzltool init` accepts `--project_name` and `--config` (JSON format matching
  the spec preamble). The `project_name` should be made available to Go
  templates via `{{.ProjectName}}`.

Always prioritize updating and consulting `ai/spec.md` when new requirements
arise.



# Fragment for setting up a standardized git workflow

## General Change Rules

- Do not change files in `//third_party` without explicit permission from the user.


## Commits

* Use "conventional commits v1.0.0" for the commit title.
- Append the following note to the last line of the commit message (below any
  summaries):
  ```
  This commit has been created by an automated coding
  assistant, with human supervision.
  ```
- Additionally, append the exact prompt used to generate the commit in full.


## Rebase and Merges

- Prefer rebasing the current branch on top of another instead of merging into
  a branch.
- E.g., to get content from `main` from the `origin` repo, use `git rebase
  --pull origin main`.
- **CRITICAL**: Always use the `--no-edit` option when running `git merge`,
  `git cherry-pick`, or `git rebase` (e.g., `git merge --no-edit`, `git rebase
  --no-edit`) to prevent interactive editors from blocking automated tasks.


## Pull Requests

- Use the `gh` utility to create the pull request.
- Use `origin/main` as a baseline for the pull request.
- Rebase the local `main` branch from `origin/main`.
- Create a new branch from `main` using the pattern
  `ai-dev-YYYYMMDD-NONCE-SHORT_TOPIC`, where `NONCE` is replaced with a random
  sequence of letters generated by executing the bundled script at
  `scripts/generate_nonce.sh`, and `SHORT_TOPIC` is replaced with an extremely
  short summary of the change.
- Push this new branch to the remote.
- Create the pull request using `gh`. The PR description must contain a summary
  of all commits between `origin/main` and the top of the branch.
- The PR description must also contain the following note at the end:
  ```
  This pull request has been created by an automated coding assistant, with
  human supervision.
  ```
- Additionally, append the exact prompt used to generate the pull request in
  full.
- The task is complete once the `gh` command finishes successfully.
- When fixing a specific Github issue, put a line "Fixes: URL", where URL
  is the link to the Github isssue being fixed.
